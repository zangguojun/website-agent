# 动态 Agent iOS UI 优化设计

| 字段 | 值 |
| --- | --- |
| 项目 | Knowledge Test iOS App |
| 日期 | 2026-05-06 |
| 状态 | 设计已确认，待实现规划 |
| 范围 | 当前 MVP 页面、设计系统、关键状态与动效 |

## 1. 设计目标

当前 App 已跑通 MVP 流程，但 UI 仍偏静态 demo。此次优化目标是把体验升级为“iOS 原生 AI 诊断工具”：

- 整体视觉以 `Native Calm` 为底色，轻盈、克制、接近 iOS 原生学习工具。
- 首页和澄清流程突出 AI 助教感，但不做重聊天产品。
- 报告页局部采用专业诊断视觉，强化可信度和结果价值。
- 所有澄清问题和测试题都按动态 Agent 推断流程设计，而不是固定问卷或固定题库。

最终方向为 `A > D > C`：

1. `A`：iOS 原生工具感优先。
2. `D`：澄清、生成和解释阶段体现 AI 助教。
3. `C`：报告页呈现专业诊断张力。

## 2. 核心体验模型

正确的产品模型不是“固定澄清问题 + 固定测试题”，而是一场由 Agent 主导的动态诊断会话。

流程如下：

1. 用户输入自然语言主题。
2. Agent 理解主题和用户意图。
3. Agent 判断是否需要澄清。
4. 如果需要，每轮只问一个高信息增益问题。
5. 用户回答后，Agent 再判断继续追问、结束澄清，或调整方向。
6. 澄清充分后，Agent 生成测试计划。
7. 用户轻量确认测试计划，或返回继续澄清。
8. Agent 根据澄清上下文动态生成测试题。
9. 用户答题。
10. Agent 生成诊断报告，解释维度、薄弱点和错误答案。

UI 采用 `对话主导 + 计划卡片`：

- 澄清阶段用聊天流表达 Agent 动态推断。
- 每个 Agent 问题可展示“为什么问”。
- 当 Agent 判断信息足够时，生成测试计划卡。
- 用户只需选择“开始生成测试”或“继续澄清”，不进入复杂配置。

## 3. 信息架构

页面保持少而清晰，但每个页面承载动态状态。

- `Home`：发起 Agent Session。
- `ClarifyChat`：动态澄清对话。
- `PlanConfirm`：测试计划确认。
- `Generating`：透明生成过程。
- `Answer`：动态测试题作答。
- `Report`：诊断结果。
- `History` / `Settings`：保持轻量。

### Home

首页不是普通表单，而是 AI 助教开场。

- 顶部小状态：例如“今天适合做一次 6 分钟诊断”。
- 大标题：`你想验证哪部分知识？`
- 助教说明卡：说明会先澄清，再生成诊断题。
- 大号输入区：支持自然语言主题。
- 建议主题：以“助教建议”呈现，而不是普通标签云。
- 主按钮：`开始诊断`。

### ClarifyChat

澄清页是动态聊天流。

- 顶部 compact header：主题名和当前状态，例如 `正在澄清范围`。
- 主区域展示 Agent 气泡、用户回答气泡和轻量上下文摘要。
- Agent 气泡包含：
  - 主问题。
  - 可选的 `为什么问` 说明。
- 底部回答区根据问题类型变化：
  - 单选。
  - 多选。
  - 开放输入。
  - `不确定` / `跳过`。

澄清问题必须来自 Agent 动态推断。UI 不能暗示这是固定步骤或固定题目。

### PlanConfirm

测试计划卡承接过程透明。

展示内容：

- 测试目标。
- 主题边界。
- 难度假设。
- 能力维度。
- 预计题量。
- 题型组成。
- 一句 Agent 解释：例如“这套计划基于你刚才的回答生成”。

用户操作：

- `开始生成测试`。
- `继续澄清`。

MVP 不提供复杂编辑，不做维度和题量配置器。

### Generating

生成中不使用普通 spinner，而展示 Agent 工作步骤：

1. 分析主题边界。
2. 划分能力维度。
3. 生成选择题。
4. 校验选项和干扰项。
5. 准备报告框架。

每一步有等待、进行、完成和失败状态。

### Answer

答题阶段必须专注，不让 Agent 干扰测试有效性。

- 顶部显示题号和进度。
- 显示当前维度。
- 一屏一题。
- 选项使用大点击区域。
- 底部按钮为 `下一题` / `提交`。
- 不显示答案解释。
- 不显示 Agent 对当前答案的反馈。

题目来源是 Agent 生成的测试计划，而不是固定题库。

### Report

报告页局部采用专业诊断视觉。

- 顶部深色分数卡：总分、掌握等级、一句话结论。
- 维度区：条形图或轻量雷达图。
- `为什么这样评估`：把澄清答案和测试维度关联起来。
- `薄弱点 TOP3`。
- 错题解析列表。
- CTA：`重新测试相近主题` / `回到首页`。

报告页要让用户理解：这份结果来自前面的澄清、计划和答题，而不是一个普通 quiz 分数。

## 4. 设计系统

### 颜色

- 背景：冷白，例如 `#F8FAFC` / `#F5F7FB`。
- 主色：系统蓝或深蓝黑，避免大面积高饱和。
- Agent 色：柔和蓝紫或青色，只用于 Agent 状态、澄清气泡和生成步骤。
- 报告强调色：
  - 绿色表示熟练。
  - 琥珀色表示薄弱点。
  - 深色卡用于总分区域。

### 字体层级

- 首页和报告使用 large title 风格。
- 正文保持 15-17pt，行高充足。
- 题干使用 22-26pt。
- 小标签使用 12-13pt 和半粗体。

### 核心组件

- `ScreenShell`：统一安全区、页面 padding 和背景。
- `PrimaryButton`：default / pressed / disabled / loading。
- `AgentBubble`：主问题和可选“为什么问”。
- `ChoiceOption`：selected / disabled / multi-select。
- `PlanCard`：目标、范围、难度、维度、题量。
- `GenerationStepList`：Agent 生成步骤。
- `QuestionCard`：题干、维度和选项。
- `ReportScoreCard`：总分和掌握标签。
- `MetricBar` / `WeaknessCard`：报告页诊断组件。

## 5. 状态设计

动态 Agent 流程必须明确表达状态，否则用户会觉得不可靠。

- `idle`：等待用户输入主题。
- `creatingSession`：创建诊断会话。
- `clarifying.streaming`：Agent 正在生成下一问。
- `clarifying.awaitingUser`：等待用户回答。
- `plan.ready`：测试计划可确认。
- `generating`：题目生成中。
- `answering`：答题中。
- `report.generating`：报告生成中。
- `error.retryable`：网络或 Agent 失败，可重试。
- `error.blocking`：无法继续，需要回首页或新建 session。

## 6. 关键动效

动效要像 iOS 原生：轻、短、服务状态。

- 页面进入：卡片轻微上移并 fade in。
- Agent 思考：小圆点 pulse 或 shimmer。
- 新气泡出现：从下方 8-12px 轻轻浮入。
- 选项点击：scale 0.98 和背景色过渡。
- 生成步骤：逐项 checkmark，当前步骤 pulse。
- 答题下一题：题卡轻微 slide 或 fade replace。
- 报告出现：分数卡先出现，维度条随后 stagger reveal。

不做：

- 游戏化积分、连击、等级。
- 重聊天输入工具栏。
- 影响答题的实时答案解释。

动效必须支持无障碍降级。减少动画时可退化为 opacity 变化。

## 7. 错误处理

错误文案必须说明当前卡在哪一步，而不是只显示“稍后再试”。

### 创建会话失败

- 位置：首页输入卡内。
- 文案：`没能创建诊断会话，请检查网络后重试。`
- 操作：`重试`。

### Agent 澄清失败

- 保留已有聊天记录。
- 在气泡流底部显示错误卡：`Agent 没能生成下一问。`
- 操作：`重试生成问题` / `返回首页`。
- 不清空用户已回答内容。

### 测试计划生成失败

- 停留在澄清页或计划页。
- 文案：`当前信息不足或生成失败，可以继续澄清后重试。`
- 操作：`继续澄清` / `重新生成计划`。

### 题目生成失败

- 停留在 `Generating`。
- 标记失败步骤，例如 `校验题目时失败`。
- 操作：`重新生成测试`。

### 答题提交失败

- 当前题答案保留。
- 错误出现在底部按钮上方。
- 操作：`重试提交`。

### 报告生成失败

- 如果已有基础分数，先展示基础结果。
- 报告解释部分可重试。
- 操作：`重新生成报告`。

## 8. 测试验收

至少验证这些用户路径：

1. 输入主题，创建 session，进入动态澄清。
2. Agent 提问，用户选择或输入，Agent 继续追问。
3. Agent 判断信息足够，显示测试计划卡。
4. 用户继续澄清，回到聊天流。
5. 用户确认计划，进入生成步骤。
6. 生成完成，进入答题。
7. 选择答案，下一题，最后一题提交。
8. 报告页展示总分、维度、薄弱点和解释。
9. 网络失败时上下文不丢失，并能重试。

自动化建议：

- 组件级：按钮、选项、计划卡、生成步骤、题卡状态。
- 页面级：澄清状态机、计划确认、答题推进。
- E2E smoke：完整走一遍 MVP 流程。
- 真机视觉验收：safe area、键盘、长文本、深浅色适配。

## 9. 实现边界

这次 UI 优化不要求一次性接入完整真实 Agent 后端，但 UI 必须按真实动态流程设计。

建议分两层实现：

1. 前端用 mock dynamic state 驱动，先跑通页面结构、状态和动效。
2. 接入已有 API / SSE，用真实 Agent 的动态澄清、计划和生成状态替换 mock 数据。

不做：

- 复杂计划编辑器。
- 用户自定义题量或维度的高级配置。
- 积分、排行榜、游戏化系统。
- 完整订阅和登录 UI 重设计。
- 报告分享页重设计。

## 10. 成功标准

完成后，App 应从“静态 demo 页面”升级为“动态 Agent 诊断会话体验”：

- 用户能感知 Agent 在动态理解主题，而不是走固定问卷。
- 澄清问题、测试计划和测试题在 UI 表达上都是动态生成的。
- 流程透明但不复杂。
- 首页、澄清、生成、答题、报告都共享一致的 iOS 原生视觉语言。
- 报告页能建立专业可信感，并与前面的澄清和测试计划形成闭环。
